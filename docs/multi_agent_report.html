Multi-Agent System

Airlines Refund Decision Maker · Gradio on :7862 · Classifier → Researcher → Analyst → Writer → Judge

1. High-level picture

The user submits a case from the browser. The Gradio app receives it, then either serves a cached/DB answer or runs the full pipeline (Classifier → Researcher → Analyst → Writer → Judge) and saves the result.

2. Request flow (step by step)

Each request goes through cache → DB → pipeline. First hit wins; only on full miss do we run all five agents.

User Question → Cache (exact / semantic)
↳ miss → DB (exact / semantic)
↳ miss → Classifier → Researcher → Analyst → Writer → Judge
→ Save (cache + Excel + DB)

User question

Form submitted (case type, flight type, ticket type, payment, accepted alternative, description). Logged as USER QUESTION block.

Cache lookup

Exact match by input hash; on miss, semantic match (embedding + cosine similarity ≥ 0.9). If hit → return cached decision and stop.

DB lookup

PostgreSQL: exact by hash, then semantic by embedding (≥ 0.9). If hit → return DB decision and stop.

Full pipeline (on cache + DB miss)

3a. Classifier (LLM) — Extracts case category, delay/bag hours, payment, key facts, etc. → case summary.
3b. Researcher (LLM + tool) — Uses search_regulations on the RAG index (PDFs in bilgiler/). Returns applicable rules.
3c. Analyst (LLM + tools) — Uses check_delay_threshold, check_baggage_threshold, calculate_refund, calculate_refund_timeline. Decides APPROVED / DENIED / PARTIAL.
3d. Writer (LLM + tool) — Produces structured JSON decision and, if approved/partial, generate_decision_letter.
3e. Judge (LLM) — Reviews the outcome; can override if something looks wrong.

Save

Append to JSON cache, export to Excel (decision_log.xlsx), insert one row into PostgreSQL (with embedding for future semantic hit). Log SUMMARY and FLOW DIAGRAM, then END OF FLOW.

4. Tools used by agents

Deterministic tools (in refund_tools.py) do the math and rule lookups; the LLMs decide when to call them.

6. Decision outcomes

The pipeline produces one of three decisions, surfaced in the UI and logs:

✅
APPROVED
Refund granted; letter generated
❌
DENIED
Does not meet DOT thresholds
⚠️
PARTIAL
Partial refund or special case

7. One-sentence story

Gradio receives your question, tries cache and PostgreSQL for a quick answer, and on miss runs a local multi-agent pipeline (Classifier → Researcher → Analyst → Writer → Judge) over your regulations and tools, then saves the result back into the cache and DB and logs everything in detail for that single question.
